## 1. Topology — one cluster, no HA

> **Single-tenant: one instance per enterprise, in the bank's own GCP project, bank-operated.**
> The delivery & operating model — tenancy, upstream↔adopter responsibilities, onboarding,
> upgrades, data ownership — is **[DELIVERY-MODEL.md](./DELIVERY-MODEL.md)**; this section is
> the **runtime topology** of one instance.

**Everything on one GKE cluster — no Cloud Run, no HA.** Web UI, serving/API/MCP, Temporal
server, ingestion workers, and the database all run in-cluster; Temporal persistence reuses
the same AlloyDB instance (separate database). **Single replicas throughout** (a deliberate
cost choice — DECISIONS 9).

| Workload                | Shape                                           |
| ----------------------- | ----------------------------------------------- |
| serving / API / MCP     | single replica, KEDA-scaled (→ 0 when idle)     |
| reasoning endpoint (TS) | own pod, scales independently (KEDA)            |
| web (Vue SPA)           | static, served behind the ingress               |
| Temporal server         | single replica; persistence in the shared DB    |
| ingestion workers       | scale-to-zero between (rare) indexing runs      |
| AlloyDB Omni            | one instance, ~2 vCPU, PVC-backed (DECISIONS 4) |

Images are built by **Podman/Buildah**, signed, and admitted only if the signature verifies
(CI-CD §3). Migrations run as a pre-deploy Job (`cmd/migrate`).

## 2. Auto scale-down

Compute auto-scales to real usage (~50%); the licence/cluster-mgmt/storage floor can't
scale (COST.md §4):

- **Stateless services** (serving/API/UI/MCP, Temporal) → **KEDA** + **cluster autoscaler**
  scale replicas/nodes to load, down to `minReplicas=0` when idle; the emptied node is
  removed. **Pre-warm** before known busy windows to hide cold starts.
- **Ingestion workers** → already scale to zero.
- **Database** → the AlloyDB Omni pod can shrink/stop when idle (PVC persists, ~1–2 min cold
  start). The **licence is a fixed floor** scaling can't remove in the reference cost model
  (DECISIONS 16; verify current vendor terms before budgeting).

> Cold starts (KEDA scale-to-zero, AlloyDB idle-stop) trade availability for cost — they are
> an explicit SLO budget line (OBSERVABILITY §4).

## 3. Reliability, backup & DR

**No HA** is a cost choice — but **durability is not optional**:

- **AlloyDB Omni:** scheduled logical/physical backups to GCS + WAL/PITR; **target RPO ≤ 24 h,
  RTO a few hours** (restore to a fresh pod; the PVC is the live copy).
- **Raw files in GCS:** immutable, **object-versioned**, tier-scoped buckets — the system of
  record; the DB is rebuildable by re-ingest + re-embed.
- **Graph/attestations** (human-promoted edges, the golden set) are the **irreplaceable**
  data — back them up first; they can't be regenerated by re-ingest.
- **Blast radius:** a node/pod loss is downtime, not data loss; cold-start + restore is the
  recovery path. Acceptable for an internal audit tool; revisit if it becomes critical
  (COST.md §6 prices an HA replica).

## 4. Ingest connectors & operational limits

Internal corpora use a **pluggable `ingest.Source`** (DATA-MODEL §1). **Where no
Graph/Azure-AD app is provisioned** (often the case — it carries setup cost + a security
sign-off), the **web crawl is the default**; Graph is an optional upgrade:

| Connector                                   | Auth                                  | Discover                                     | Notes                                                                                                                                                  |
| ------------------------------------------- | ------------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **SharePoint web crawl** _(common default)_ | **AD account** sign-in (no Graph app) | authenticated crawl + watermark              | no app/extra spend; authoritative metadata from the parsed **doc-control block + per-source config** (web-UI columns a weak hint only — DATA-MODEL §2) |
| **SMB file share**                          | AD service account / share creds      | directory walk + content hash                | metadata from filesystem + side-car manifest                                                                                                           |
| **SharePoint / Graph** _(optional)_         | Azure AD app (client-credentials)     | Graph **delta query** → per-source watermark | only real edge is incremental sync; its columns = uploader, not signer (DATA-MODEL §2) — only if provisioned                                           |

Operational caveats specific to the web crawl (the default):

- **MFA / Conditional Access** often blocks a headless AD login — needs an **approved,
  scoped read-only service account** (IP-allowlisted), a security sign-off (DECISIONS 13).
- **Brittleness:** HTML scraping + session/pagination handling; honour throttling (429 +
  `Retry-After`), bound per-site concurrency.
- **Metadata gap:** web/SMB give less structure than Graph — recover signer/owner/version/
  effective_date from the **doc-control header + side-car manifest** (DATA-MODEL §1/§2).

Plus, common to all connectors:

- **Doc AI / Vertex quotas** (pages/min, requests/min, tokens): batch where supported,
  backpressure via Temporal retry policies (exponential backoff + jitter); never hammer.
- **Idempotent activities** keyed by content hash + `ingest_run_id` so retries don't
  duplicate chunks or re-judge unchanged clauses.

## 5. Auth, secrets & region

- **AuthN/Z:** OIDC/IAM at the ingress; per-corpus access tier resolved per request → bound
  to the DB session role (RLS, DATA-GOVERNANCE §2/§7). Internal MCPs + UI are
  access-controlled (NOT public).
- **Secrets** (Azure AD app / AD account / SMB creds, DB creds) in **Secret Manager**,
  mounted at deploy — never baked into images (CI-CD §5).
- **Region pinning** (DECISIONS 17): configure GCS, AlloyDB, Vertex, and backups in the bank's
  selected GCP region(s). This is deploy config, not a project gate.
